## Настройка личности в git (один раз, локально для этого репо)

Не трогай глобальный `git config` — задай личность **только внутри папки
проекта**, чтобы не сломать identity в других репозиториях на этой машине:

```
cd путь/к/NewBurmolda
git config user.name "Tonics42"
git config user.email "artembigrussia@gmail.com"
```

## Правило соавторства: коммить ВДВОЁМ

Проект ведут два человека через ИИ-ассистентов — поэтому **каждый коммит
должен указывать обоих**, даже если фактически правки внёс один. Это честно
отражает, что решения и код проходят через обе стороны (просмотр, идеи,
правки друг друга).

- **Коммитер** (`user.name`/`user.email`) — тот, с чьей машины реально уходит
  коммит (см. настройку выше).
- **Co-authored-by** — второй человек, трейлером в конце сообщения коммита.

Если коммитит Tonics42 — добавляй:
```
Co-authored-by: Arkorij <3yu77op@gmail.com>
```

(Если однажды коммитит Arkorij — трейлер обратный:
`Co-authored-by: Tonics42 <artembigrussia@gmail.com>`.)

## Формат сообщения коммита

- Заголовок — **повелительное наклонение**, коротко, по-русски, без точки в конце:
  `Добавить крафт у кузнеца`, а не `Добавлен крафт` или `crafting added`.
- Заголовок описывает **что изменилось**, тело (через пустую строку) — **зачем**:
  какую проблему решает, что стало возможно, почему выбран такой подход.
  Не пересказывай diff построчно — это видно и так.
- Без «мусорных» коммитов вида `fix`, `wip`, `правки`. Если правка мелкая —
  всё равно опиши суть в одну строку.

Пример:
```
Добавить крафт у кузнеца: два одинаковых предмета -> тир выше

Замыкает экономику: добыча -> продажа -> покупка -> крафт. Раньше
скопленные ресурсы девать было некуда после закупки базового шмота.

Co-authored-by: Arkorij <3yu77op@gmail.com>
```

## Перед коммитом — обязательный чек-лист

1. **Прогони headless-тесты**, они должны быть полностью зелёными:
   ```
   tools\godot\Godot_v4.7-stable_win64_console.exe --headless --path . -- --test
   ```
   Если тест упал — почини или добавь новый тест под свою фичу
   (`tests/run_tests.gd`), **не коммить в красном состоянии**.
2. Если менял GDScript-файл с `class_name` (новый скрипт) — один раз прогони
   `--import`, чтобы Godot пересобрал реестр глобальных классов:
   ```
   tools\godot\Godot_v4.7-stable_win64_console.exe --headless --path . --import
   ```
3. Проверь `git status` — не должно попадать:
   - `tools/godot/` (портативный движок, скачивается отдельно, в `.gitignore`)
   - `.godot/` (локальный кэш импорта Godot, тоже в `.gitignore`)
   - `_shots/` / `_shot_*.png` (мусор от отладочного скриншотера)
   - если что-то из этого попало в `git status` — значит `.gitignore` не
     сработал (например, файл уже был закоммичен раньше) — не добавляй руками.
4. `*.uid` рядом со скриптами — это НЕ мусор, Godot 4 хранит в них стабильные
   ссылки на ресурсы. Их нужно коммитить вместе с `.gd`-файлом.

## Безопасность истории — как в обычном проекте

- **Не форси пуш** (`push --force`) в `main`, особенно если Arkorij мог успеть
  запушить параллельно — сначала `git pull`, разреши конфликты, потом пуш.
- **Не переписывай чужие коммиты** (`commit --amend`, `rebase -i`) после того,
  как они попали на GitHub — это ломает историю у второго разработчика.
- Каждая правка — отдельный коммит с чёткой темой, а не одна огромная свалка
  за неделю работы. Так проще откатить конкретную фичу, если она сломает игру.
- Если делаешь рискованную/крупную фичу (например, трогаешь `battle.gd` или
  генерацию мира) — можно сначала завести ветку (`git checkout -b feature-x`),
  а слить в `main` через Pull Request, чтобы Arkorij успел глянуть до мержа.
  Для мелких и проверенных тестами правок это необязательно — можно прямо в `main`.
